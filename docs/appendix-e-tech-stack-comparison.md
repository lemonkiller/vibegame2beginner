# 附录E 技术栈对比

> 选技术栈就像选车：有人需要越野车翻山越岭，有人只需要一辆省油的代步车。选错了，不是车不好，是不适合你。

**数据时效性说明**：本附录中的功能特性、包体大小、定价等信息截至 **2026年4月**。技术栈迭代频繁，建议查看各项目官网获取最新信息。

---

## 为什么需要了解不同技术栈

你可能会有疑问：这本手册不是教 LÖVE2D 吗，为什么还要看别的？

原因很简单——**LÖVE2D 不是唯一的选择，也不一定是你的最佳选择**。就像本手册选 LÖVE2D 是因为它特别适合"零基础 + AI编程"这个组合，但如果你想做 3D 游戏、想上 Steam 卖钱、或者你就是喜欢在浏览器里做东西，别的技术栈可能更合适。

了解全貌之后做选择，比一开始就蒙着头往前冲要好得多。

### 本手册为什么选 LÖVE2D

- **Lua 语法最简单**，跟写英语差不多，零基础友好
- **AI 对 Lua 的代码生成质量好**，尤其是 Claude，生成 LÖVE2D 代码又快又准
- **包体极小**，一个完整小游戏打包后不到 5MB
- **专精 2D**，不用学一堆跟 2D 无关的东西
- **免费开源**，不花钱，没有商业限制

> **怎么看这张表？**
>
> - **训练数据量**：AI 看过的该语言代码越多，生成质量越高。Python 和 JavaScript 因为是互联网最常见的语言，AI 对它们最"熟悉"。GDScript 和 GML 只在各自的引擎里用，AI 见过的代码少，所以生成质量会差一些。
> - **代码简洁度**：语言本身越简洁，AI 需要生成的代码越少，出错概率越低。Lua 和 Python 在这方面表现最好——写同样功能的代码，它们需要的行数可能只有 C++ 的三分之一。
> - **AI工具生态**：有没有专门给这个技术栈用的 AI 插件、AI 编程模板、社区 AI 经验帖。Unity 和 Web 技术栈在这方面最成熟。
> - **项目结构透明度**：AI 修改代码时，能不能一眼看出"这个功能在哪个文件里"。LÖVE2D 这种框架项目结构简单透明，AI 改起来很顺利；Unreal 那种包含几百个配置文件的大型引擎，AI 经常找不到该改什么。

> **特别说明——GDScript 的问题**：Godot 的 GDScript 语法虽然简单，但 AI 对它的理解远不如 Python 或 JavaScript。因为 GDScript 只有 Godot 在用，训练数据量有限。实际测试中，用 C# 写 Godot 代码比用 GDScript 让 AI 生成更靠谱——C# 的训练数据量远大于 GDScript。这也是 Godot AI 编程友好度只有 3 星的原因。

## 三大类详细说明

### 游戏引擎——"自带厨房的精装房"

游戏引擎就像精装房：厨房、卫生间、家具都给你配好了，拎包入住。Unity、Godot、Unreal、GameMaker、Defold、Cocos Creator 都属于这一类。

它们有可视化编辑器——一个可以拖拖拽拽排列场景、调整参数的界面。好处是"所见即所得"，坏处是你得学怎么用这个编辑器，而编辑器的学习成本有时候比学写代码还高。

对于 AI 编程来说，引擎类有个天然劣势：**AI 改代码的时候看不到编辑器里配了什么**。比如 Unity 场景里有个物体叫"Player"，AI 是看不到的——AI 只能看到代码文件。如果 AI 需要知道"Player 上挂了哪些组件"，你得手动告诉它，或者让 AI 去找配置文件。这就增加了沟通成本。

### 纯代码框架——"自己装修的毛坯房"

纯代码框架就像毛坯房：给你墙和水电，装修靠自己。LÖVE2D、Raylib、Pygame、LibGDX、MonoGame、Bevy 都属于这一类（Construct 3 虽然是事件驱动不需要写代码，但同样没有可视化场景编辑器让 AI 直接操控）。

它们没有可视化编辑器，所有东西都靠代码搞定。听起来好像更难？但对于 AI 编程场景来说，**这反而是优势**——因为 AI 最擅长的就是读写代码。你告诉 AI "加一个会跳的角色"，AI 就直接写代码加一个，不需要纠结"编辑器里该点哪个按钮"。

LÖVE2D 在这个类别里特别适合新手的原因：Lua 语法极其简单（接近英语），项目结构一目了然（一个 `main.lua` 搞定一切），AI 对 Lua 的代码生成质量很好。

### Web技术栈——"搭在商场里的摊位"

Web 技术栈就像商场里的摊位：不用自己建房子，商场（浏览器）帮你管水电物业，你只想怎么经营。Phaser、PixiJS、Three.js、Kaboom.js 都属于这一类。

最大的好处是**不需要安装任何东西**——写完代码，浏览器打开就能看。发布也超简单：扔到网上，给个链接就能让全世界的人都能玩到。

但 Web 技术栈也有局限：性能不如原生应用，做大型游戏会吃力；而且 JavaScript 这个语言虽然 AI 很熟悉，但它的"灵活"（或者说"混乱"）有时候会让 AI 生成风格不一致的代码。

### AI 编程场景下的特别推荐

> 这一段是本附录的重点：**在 2026 年，AI 是写代码的主力，你是指挥官**。所以选技术栈的时候，要优先考虑"AI 用得顺不顺手"。

| 场景 | 推荐 | 不推荐 | 说明 |
|------|------|--------|------|
| 零基础 + AI 写全部代码 | LÖVE2D、Pygame | Unreal、Bevy | AI 对 Lua/Python 生成质量最好，C++/Rust 差很多 |
| 用 AI 做网页小游戏 | Phaser、Kaboom.js | PixiJS（它不是游戏框架）| Phaser 有完整的游戏循环，AI 生成代码可以直接跑 |
| 用 AI + 引擎做 2D | Godot（但用 C# 而不是 GDScript） | GameMaker | GDScript 训练数据少，AI 生成不如 C# 靠谱 |
| 用 AI + 引擎做 3D | Unity | Unreal | Unity 的 C# AI 生成质量远好于 Unreal 的 C++ |
| 纯 AI 对话做游戏（零代码） | Web 方案或 LÖVE2D | Unreal | 代码越少、结构越透明，AI 越不容易犯迷糊 |

**核心原则**：AI 编程场景下，**语言训练数据量 > 引擎功能丰富度**。一个 AI 对语言很熟悉但功能少一点的框架，比一个 AI 不太熟但功能很多的引擎，出活更快。
