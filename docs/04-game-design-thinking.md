# 第4章 游戏设计思维——从“怎么做”到“做什么”

> AI什么都会做,但你说不清楚想要什么,做得再好也不是你想要的。

## 一个尴尬的事实

以前做游戏,80%的时间花在"怎么做"上--怎么写碰撞检测、怎么画图、怎么播声音。20%的时间花在"做什么"上--游戏好玩不好玩、玩家会有什么感受。

AI把这个比例倒过来了。现在"怎么做"几乎零成本--你说"加个碰撞检测",AI 30秒给你写好。真正的瓶颈变成了:

**你到底想要什么样的游戏?**

这就像你站在一家自助餐厅前。以前的问题是你不会炒菜(怎么做),所以只能点菜单上有的。现在AI是大厨,什么都会做,问题变成了--你今天想吃什么?如果你说不出来,大厨做的再好也不是你想要的。

---

## "做什么"的三个层次

想清楚"做什么",可以分三个层次来想:

### 第一层:核心体验一句话

用一个句子描述:**玩家在干什么,感觉如何。**

不需要详细,不需要完整,只需要一个画面感:

| 核心体验一句话 | 画面感 |
|--------------|--------|
| "玩家在黑暗洞穴里摸索,每一步都可能踩到陷阱" | 紧张×探索 |
| "玩家弹球消除砖块,连击的时候超爽" | 爽快×策略 |
| "玩家控制食物躲避蛇,像捉迷藏" | 新奇×紧张 |

这一句话就是你整个游戏的方向盘。后面所有设计决策都回来对照这句话--加的东西是否强化了这个核心体验?如果不是,别加。

### 第二层:核心循环

玩家**重复做**的事情是什么?

```
贪吃蛇的核心循环:移动 → 吃 → 变长 → 更难移动
打砖块的核心循环:反弹 → 消除 → 得分 → 继续反弹
RPG的核心循环:  探索 → 对话/战斗 → 获得信息/道具 → 去新地方
```

核心循环是游戏的"心跳"。一个好游戏只需要一个强循环,不需要五个弱循环。

### 第三层:一个不同点

你的游戏和同类游戏有什么不同?**一个就够了。**

- 贪吃蛇但是蛇能穿墙 → 穿墙贪吃蛇
- 贪吃蛇但是你控制食物 → 反向贪吃蛇
- 贪吃蛇但是只能跟着节拍转向 → 节奏贪吃蛇

不需要发明全新的品类。改一个变量,就是一个新游戏。

## 体验倒推设计法

不要从"功能"开始想,从"感受"开始想:

1. 我想让玩家感到___
2. 需要什么机制制造这种感受
3. 告诉AI实现这个机制

| 我想要的感受 | 制造这种感受的机制 | 给AI提示词的关键词 |
|------------|----------------|------------------|
| 紧张 | 倒计时、追逐者 | "倒计时""追踪AI" |
| 爽快 | 连击、爆炸效果 | "连击系统""粒子爆发" |
| 探索欲 | 隐藏房间、随机地图 | "房间发现""地图随机" |
| 成就感 | 解锁、升级 | "技能树""收藏系统" |
| 温馨 | 柔和色彩、慢节奏 | "暖色调""慢速移动" |

**这个表格是游戏设计最重要的工具**--它把模糊的"感觉"翻译成了AI听得懂的"机制"。

## 约束是创意的朋友

第3章提到了“约束驱动法”,这里从设计角度再说一遍:

**你给自己加的约束越多,创意反而越集中。**

Game Jam(游戏开发马拉松)的经验:48小时+给定主题 = 创意爆炸。那些做了最有趣游戏的队伍,往往不是功能最多的,而是约束最清晰的。

实操模板--给自己定2-3个约束:

```
我的游戏:
- 只有 ___ 个按钮(操作约束)
- 只用 ___ 种颜色(视觉约束)
- 场景只有 ___ 个(规模约束)
```

例子:
- "只有1个按钮,只用4种颜色,1个场景" → 你会被迫想想怎么在极简中创造乐趣
- "3个按钮,黑白两色,5个关卡" → 你会被迫让关卡设计本身说话

## 设计文档极简模板

你不需要写几十页的GDD(Game Design Document)。AI时代,一张便签就够了:

```
游戏名称:___
一句话描述:玩家___,感觉___
核心循环:___ → ___ → ___
约束:只有___种颜色 / ___个按钮 / ___个关卡
和同类游戏的不同:___
```

**填一个例子**:

```
游戏名称:反向贪吃蛇
一句话描述:玩家控制食物躲避蛇,感觉像捉迷藏
核心循环:躲避 → 蛇变长 → 空间变小 → 更紧张
约束:3种颜色、1个按钮(左右转向)
和同类游戏的不同:你是食物不是蛇
```

写完这张便签,你就可以拿着它去找AI了--这就是CARE框架中C(Context)最好的素材。

## 新手最常见的三个设计误区

### 误区一:一上来就想做MMO

"我要做一个万人在线的开放世界RPG!"

**现实**:这类游戏需要几十人团队做几年。你一个人+AI,先做一个核心循环试试。

**修正**:把大想法压缩到最小可验证版本。"开放世界RPG" → "一个房间、一个NPC、一场战斗"。

### 误区二:功能堆砌 = 好玩

"我要加技能树、装备系统、宠物系统、坐骑、公会......"

**现实**:好玩来自核心循环的打磨,不是功能数量。一个打磨好的贪吃蛇比一个半成品MMO好玩一万倍。

**修正**:先做核心循环,玩10遍。如果核心循环不有趣,加再多功能也没用。

### 误区三:必须完全创新

"我要做一个前所未有的游戏!"

**现实**:完全创新的风险极高,连专业团队都不敢轻易尝试。改一个变量就是创新--"贪吃蛇但是你控制食物"已经够新鲜了。

**修正**:找一个你喜欢的游戏,改一个变量。改角色、改视角、改规则、改节奏、改目标。
