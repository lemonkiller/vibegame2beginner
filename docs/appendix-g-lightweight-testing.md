# 附录G 轻量级测试入门

> "AI时代，测试比你想象的重要得多。"

## 引言：为什么AI时代需要测试？

说句大实话——代码是 AI 写的，你真的确定它对吗？

AI 写代码很快，但它也会犯错。有时候少写一个条件判断，有时候边界情况没考虑，有时候重构之后悄悄改坏了什么。你盯着屏幕看不一定看得出来，但**测试能替你盯着**。

打个比方：测试就像体检。你可能觉得自己身体挺好，但只有验了血、拍了片，才能确认真的没问题。测试就是给代码做体检——**它能告诉你，哪里出问题了，而且是在玩家发现之前。**

来看看 Void Balls 这个案例：它有 **88 个测试文件**。为什么这么多？因为 AI 在重构代码的时候，测试就是安全网——代码改坏了，测试立刻告诉你哪里坏了，不用等到游戏跑起来才发现"咦怎么分数不对"。

如果你还在犹豫"我一个新手，测试是不是太难了"——别担心，这篇就是为你写的。我们从最最基础的地方开始。

---

## 一、测试是什么？（5分钟理解）

别被"测试"这个词吓到，它的核心逻辑简单到你可能已经在用了。

想象你的游戏里有一个计分函数：

```lua
function add_score(n)
    score = score + n
end
```

每次调用 `add_score(10)`，分数就加 10。看起来没问题是吧？但你怎么**确认**它没问题？

测试就是这样做的：

1. 调用 `add_score(10)`
2. 检查 `get_score()` 是不是等于 10
3. 如果等于——✓ 没问题
4. 如果不等于——✗ 有 bug！

就这么简单。测试 = **调用代码 + 检查结果**。

你之前可能做过这样的事：改完代码，运行游戏，手动试一下"加10分对不对"。那其实就是手动测试。我们现在要做的，只是**把手动检查变成自动检查**——让程序替你验证，不用每次自己盯着看。

## 二、LÖVE2D 的测试怎么做？

好消息：你不需要安装任何测试框架，不需要学什么复杂工具。

最简单的方式就是写一个独立的 Lua 脚本，直接运行它。

### 一个最小的测试文件

```lua
-- tests.lua 运行方式：lua tests.lua（不是 love .）
local tests_passed = 0
local tests_failed = 0

local function assert_equal(actual, expected, message)
    if actual == expected then
        tests_passed = tests_passed + 1
        print("  ✓ " .. message)
    else
        tests_failed = tests_failed + 1
        print("  ✗ " .. message .. " (期望 " .. tostring(expected) .. "，实际 " .. tostring(actual) .. ")")
    end
end

-- 测试分数系统
local score = 0
local function add_score(n) score = score + n end
local function get_score() return score end

assert_equal(get_score(), 0, "初始分数应该是0")
add_score(10)
assert_equal(get_score(), 10, "加10分后分数应该是10")
add_score(5)
assert_equal(get_score(), 15, "再加5分后分数应该是15")

print("\n结果：" .. tests_passed .. " 通过，" .. tests_failed .. " 失败")
```

### 运行它

打开终端，进入项目目录：

```bash
lua tests.lua
```

注意：这里用的是 `lua tests.lua`，**不是** `love .`。因为测试不需要打开游戏窗口，我们只是在检查逻辑对不对。

你会看到这样的输出：

```
  ✓ 初始分数应该是0
  ✓ 加10分后分数应该是10
  ✓ 再加5分后分数应该是15

结果：3 通过，0 失败
```

全绿勾？恭喜，你的分数系统没问题。

如果有个测试失败了，你会看到：

```
  ✗ 加10分后分数应该是10 (期望 10，实际 11)
```

一目了然——你知道哪里错了，马上去查。

### 怎么组织测试文件？

推荐这样放：

```
你的游戏/
├── main.lua
├── score.lua        ← 游戏逻辑
└── tests/
    └── test_score.lua   ← 对应的测试
```

每个模块一个测试文件，简单清晰。

## 三、哪些地方值得写测试？

新手不需要测试所有代码——那会让人崩溃。**只测试最核心的：**

### 1. 计分逻辑

分数加减对不对？重置后是不是回到 0？加负数会怎样？

```lua
-- 分数测试
add_score(10)
assert_equal(get_score(), 10, "加分正常")

reset_score()
assert_equal(get_score(), 0, "重置后应该是0")

add_score(-5)
assert_equal(get_score(), -5, "加负数后分数可以为负")
```

### 2. 碰撞检测

两个物体重叠了，检测到了吗？没重叠，不会误判吧？

```lua
-- 碰撞测试
assert_equal(check_collision(0,0,10,10, 5,5,10,10), true, "重叠的物体应该碰撞")
assert_equal(check_collision(0,0,10,10, 100,100,10,10), false, "远离的物体不该碰撞")
```

### 3. 游戏状态转换

游戏从"开始"变到"进行中"，再变到"结束"，流程对不对？

```lua
-- 状态测试
start_game()
assert_equal(get_state(), "playing", "开始后应该是进行中")

end_game()
assert_equal(get_state(), "gameover", "结束后应该是gameover")
```

### 4. 边界情况

那些"一般不会碰到，但碰到了就可能炸"的情况：

- 分数能是负数吗？
- 蛇能反向走吗？（贪吃蛇经典 bug）
- 玩家 HP 为 0 时再受伤会怎样？
- 空列表上调用 `#list` 返回什么？

```lua
-- 边界测试
add_score(-999)
assert_equal(get_score() >= 0, true, "分数不应该为负数")  -- 如果你有这个规则

set_direction("up")
set_direction("down")  -- 蛇不能直接反向
assert_equal(get_direction(), "up", "蛇不应该直接反向走")
```

### 一句话原则

**如果这段代码错了，游戏就会出明显问题——那就值得写测试。** 如果错了也无所谓，可以不写。

## 四、让 AI 帮你写测试的提示词

你不需要自己想所有测试用例——让 AI 帮你想。

把你的代码给它，然后说：

> 这是我的评分系统代码：
>
> ```lua
> （粘贴你的 score.lua 代码）
> ```
>
> 请为它写一组测试用例，覆盖：
> 1. 正常情况（加分、减分、重置）
> 2. 边界情况（分数为0时再减分、分数上限溢出等）
>
> 使用 assert_equal 风格，输出可以直接运行的 lua 测试文件。

AI 会给你生成测试代码。你检查一下逻辑对不对，然后跑一遍就行。

**但要注意**：AI 生成的测试也可能有 bug。所以：
- 看一眼测试写的对不对
- 跑一遍，检查结果是不是你预期的
- 如果测试通过了但你直觉觉得不对，仔细看看测试逻辑

测试代码也是代码，也需要你过目。

## 五、测试和 Game Feel 的关系

你可能会想："我关注 Game Feel 了，还需要测试吗？"

**答案是：两个都要。**

| | 测试 | Game Feel |
|---|---|---|
| 回答的问题 | 功能对不对？ | 感觉好不好？ |
| 检查方式 | 自动断言 | 手动试玩 |
| 保障的是 | 不会出 bug | 不会无聊 |

测试保障**底线**——代码至少是对的。Game Feel 保障**上限**——代码对了之外，还得好玩。

缺了测试：重构时心里慌，改完不知道哪里炸了。
缺了 Game Feel：功能都对，但玩家玩着没感觉。

就像一辆车：测试是刹车和方向盘，让你安全；Game Feel 是座椅和音响，让你舒服。**两者缺一不可。**

## 六、什么时候不需要测试？

测试不是越多越好，有些地方写了也是浪费时间。**大胆跳过这些：**

### 1. 一次性原型

"我就试试这个想法行不行"——写完可能明天就扔了，测它干嘛？把时间花在验证想法上。

### 2. 纯视觉效果代码

画个圆、画个渐变、调个动画速度——这些代码看得见效果就行，不需要断言"这个圆的半径是不是 50"。**你的眼睛就是测试。**

### 3. 你非常确定正确的简单逻辑

如果是一行 `return x + y` 这种，你确定不用测。时间有限，优先测那些**又复杂又重要**的部分。

### 总结成一句话

**测试是用来保护你不想出错的东西的。** 如果出错了无所谓，就不用测试；如果出错了你会心疼，就写一个。

---

## 小结

- 测试 = 调用代码 + 检查结果，就这么简单
- 不需要框架，一个 `lua tests.lua` 就够了
- 只测试最核心的：计分、碰撞、状态转换、边界情况
- 让 AI 帮你写测试，但自己过目一眼
- 测试保底线，Game Feel 保上限，两个都要
- 不该测的地方就别测，别给自己找麻烦

在 AI 帮你写代码的时代，测试是你最重要的安全网。它不复杂，不用怕，**从测一个函数开始就好。**
