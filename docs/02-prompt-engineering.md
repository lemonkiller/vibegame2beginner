# 第2章 学会和AI说话

> 编程的本质不是写代码，而是把你想做的事情说清楚。AI时代更是如此——谁描述得越清楚，谁得到的代码就越好。

## 核心原则：像点菜一样点代码

想象你在餐厅点菜。你说"来个菜"，厨师不知道你要什么。但你说"来一份宫保鸡丁，微辣，少油，花生多一点"，厨师就明白了。

跟AI要代码也是一样：

| 你说的 | AI的理解 | 结果 |
|--------|---------|------|
| "做一个游戏" | 什么样的游戏？ | 你得到一个随机的东西 |
| "做一个贪吃蛇游戏" | 哦，蛇吃食物那个 | 跑起来了，但细节可能不对 |
| "做一个贪吃蛇游戏，用方向键控制蛇的移动，吃到食物蛇变长、分数加1，撞到墙壁或自己就游戏结束" | 清清楚楚 | 这就是你想要的游戏 |

**经验法则：如果你能让一个完全不懂电脑的朋友听懂你的描述，AI也能听懂。**

## 提示词四要素

一个好的提示词包含四个部分，我们管它叫 **CARE**：

| 要素 | 含义 | 例子 |
|------|------|------|
| **C**ontext 上下文 | 我在做什么项目 | "我在用 LÖVE2D 做一个贪吃蛇游戏" |
| **A**ction 行动 | 我要AI做什么 | "帮我添加一个开始界面" |
| **R**equirement 要求 | 具体的要求是什么 | "界面居中显示游戏标题和'按空格开始'的提示" |
| **E**xample 示例 | 我想要的效果大概长什么样 | "类似经典诺基亚手机的贪吃蛇开始画面" |

不需要每次都写齐四个，但写得越多，AI给的代码越准确。

## 常见提示词模式

### 模式一：从零创建

当你想让AI帮你创建一个新功能时：

```
我在用 LÖVE2D 做一个贪吃蛇游戏。帮我创建游戏的开始界面，要求：
1. 屏幕中央显示游戏标题"贪吃蛇"
2. 标题下方显示"按空格键开始游戏"
3. 背景是深绿色的
4. 文字用白色80号字体
```

### 模式二：修改现有代码

当你想改已有功能时，把现有代码贴给AI：

```
这是我现在的贪吃蛇代码（见下方）。我想修改它：
- 现在蛇撞墙就死了。我想改成蛇从另一边出来（穿墙模式）。
- 只需要修改碰撞检测的部分，其他不要动。

当前代码：
--- main.lua ---
（这里粘贴你的代码）
```

### 模式三：修复Bug

当你遇到错误时，把错误信息完整地给AI：

```
我在运行 LÖVE2D 游戏时遇到以下错误：

错误信息：
main.lua:45: attempt to index field '?' (a nil value)

我的代码（见下方）和出错的那行（第45行）：
--- main.lua ---
（这里粘贴你的代码）

请帮我找出问题并修复。
```

### 模式四：理解代码

当你看不懂AI生成的代码时：

```
请用简单的话解释下面这段代码做了什么，就像在给一个完全不懂编程的人讲解：

（粘贴代码）

请逐行或逐段解释，用生活化的比喻。
```

## 进阶技巧

### 1. 分步骤，不要一口气要一个完整的游戏

AI一次能生成的代码有限，复杂项目要拆开做。

**不好的做法**：
> 帮我做一个完整的RPG游戏，有地图探索、战斗系统、物品栏、对话系统、存档功能

**好的做法**：
> 第一步：帮我做一个角色能在地图上移动的框架，暂时不需要其他功能

做完第一步，测试通过后，再要第二步：
> 第二步：在之前代码的基础上，添加一个简单的敌人。角色碰到敌人时，弹出战斗界面

依此类推。每一步都要先确认上一步能运行。

### 2. 每次改完都测试

**这是最重要的一条规则**。AI改完代码后，立刻运行看看效果：

```
改完代码 → 运行游戏 → 看效果
    ↓                         ↓
  效果对了                效果不对
    ↓                         ↓
  继续下一步              把问题告诉AI，让它修
```

不要攒着一堆改动不测，等你想测的时候，已经不知道哪里出了问题。

### 3. 提供上下文，不要只给一句话

当你在 Cursor 中问AI时，它能看到你打开的文件。但如果你用的是网页版AI，记得把相关代码一起贴过去：

```
我在做 LÖVE2D 贪吃蛇游戏。这是我目前的完整代码：

--- main.lua ---
（粘贴全部代码）

--- conf.lua ---
（粘贴全部代码，如果有的话）

我想添加一个暂停功能：按 P 键暂停/继续游戏。
```

### 4. 用"不要"来限制AI

有时候明确告诉AI"不要做什么"比说"要做什么"更有效：

```
帮我添加一个游戏结束界面，要求：
- 显示"游戏结束"和最终分数
- 按 R 键重新开始
- 不要改变现有的游戏逻辑
- 不要添加任何新的文件，只在 main.lua 中修改
```

### 5. 让AI解释它的修改

当你不确定AI做了什么改动时：

```
你刚才的修改改了哪些地方？请用列表告诉我每个修改的原因，我需要理解一下。
```

## 典型错误示范

### 错误1：太模糊

> 做个好玩的游戏

AI不知道你要什么，会随便给你一个东西。

### 错误2：一次要太多

> 做一个有10种敌人、5个Boss、装备系统、技能树、地图传送...

AI会努力做，但做出来的东西大概率跑不起来，因为太复杂了。

### 错误3：不说技术栈

> 帮我写一个贪吃蛇

AI可能给你Python版本、JavaScript版本、C++版本......你得说"用 LÖVE2D 和 Lua"。

### 错误4：不贴代码直接要改

> 我的贪吃蛇出Bug了，蛇不动了，帮我修

AI看不到你的代码，没法修。你得把代码和错误信息贴过去。

### 错误5：改完不测继续改

> 刚才的改好了，现在再加X功能，再加Y功能，再加Z功能...

三次改动叠在一起，任何一个出错你都找不到是哪次改坏的。一次改一个，改完就测。

## 游戏开发专用迭代方法论

社区（HowWorks、Pikoo、r/aigamedev 等）最强调的一个理念：**不要一次性要一个完整游戏，而是分层构建。每层做完测完再做下一层。**下面是从社区最佳实践中总结出的四套核心方法论。

### 一、分层构建法（Layer-by-Layer）

社区共识：不要一次性要一个完整游戏，而是分层构建。每层做完测完再做下一层。

分层模型：

```
第1层：核心机制（Core Mechanic）  → 角色能动、能跳
第2层：挑战系统（Challenge）       → 有敌人/障碍、能输赢
第3层：进度系统（Progression）      → 分数、关卡、解锁
第4层：Game Feel（Juice）          → 震动、粒子、音效
第5层：打磨与平衡（Polish）        → 难度曲线、bug修复
```

每层都有对应的提示词风格。以第1层为例，提示词应该极度聚焦：

> "我在用 LÖVE2D 做一个平台跳跃游戏。第一步，我只需要角色能在屏幕上左右移动和跳跃。暂时不需要其他任何功能——不需要敌人、不需要计分、不需要美术。只要一个方块能动就行。"

你会发现，这段提示词明确说了**不要什么**（不需要敌人、不需要计分、不需要美术），这和前面讲的"用'不要'来限制AI"是一样的思路——约束越清晰，AI输出越准确。

---

### 二、约束驱动法（Constraint-Driven）

来自 Void Balls 案例的教训：**约束创造凝聚力，不是限制创意。**

常见的约束维度：

- **调色板约束**："所有颜色只能从这7种里选"
- **操作约束**："只能用2个按钮"
- **时间约束**："48小时内做完"（Game Jam策略）
- **规模约束**："只做1个关卡"

为什么约束反而让AI输出更好？因为AI在约束范围内生成的代码和素材更一致。没有约束时，AI每次生成可能风格跳跃、逻辑矛盾；有了约束，AI的每一次输出都被限定在一个统一的框架里，各部分之间自然更协调。

**实战建议**：在写提示词之前，先给自己定2-3个约束。哪怕只是"只有3种颜色"和"整个游戏1个文件"，都比没有约束强。

---

### 三、测试驱动迭代法（Test-Then-Next）

社区经验的铁律：**改完不测 = 浪费**。

迭代流程：

```
写提示词 → AI生成代码 → 运行测试 → 
  ├─ 通过 → 提交Git → 下一个功能
  └─ 失败 → 描述问题给AI → 修复 → 重新测试
```

关键规则：

1. **一次只做一个可验证的改动**
2. **每次改动后必须在30秒内能验证**（运行游戏看效果）
3. **验证通过后立刻 git commit**
4. **如果改动需要超过3轮AI修复才成功，停下来重新思考方案**

规则4尤其重要——当你发现自己和AI来回修了3轮还没搞定，往往意味着你的方案本身有问题，而不是AI理解错了。这时候应该退一步，换个思路重来，而不是继续在错误的方向上纠缠。

---

### 四、提示词质量分级

社区总结的提示词等级，从差到好：

| 等级 | 例子 | 通常结果 |
|------|------|--------|
| **F级**："做个游戏" | AI自己猜 | 随机产物 |
| **C级**："做个贪吃蛇" | 基本可运行 | 缺细节 |
| **B级**："做个贪吃蛇，方向键控制，穿墙，加速" | 大部分对 | 需迭代 |
| **A级**："用LÖVE2D做个贪吃蛇，800x600窗口，网格20x20，方向键控制，蛇每吃一个食物速度+5%，撞墙游戏结束，显示分数和最高分，暗绿色背景" | 一次对80% | 少量微调 |

回头看前面"核心原则"里的点菜比喻——F级就像对服务员说"来个菜"，A级就像"来一份宫保鸡丁，微辣，少油，花生多一点，配米饭"。描述越具体，结果越接近你的预期。

> **结论：花10分钟写详细提示词，省3小时来回修复。**
