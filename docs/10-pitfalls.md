# 第10章 避坑指南

> 别人踩过的坑，你不用再踩。这份指南是无数次"为什么又不行了"的经验总结。

---

## 坑位一：AI生成的代码跑不起来

### 现象
你把AI给的代码复制进去，运行 `love .`，弹出错误或者窗口闪退。

### 原因
- AI可能给了一个过时版本的 LÖVE API
- AI可能用了 LÖVE 不支持的 Lua 库
- AI可能漏掉了一些依赖的函数

### 解决方案

1. **看错误信息**：终端里会打印错误行号和错误类型。把完整的错误信息复制给AI：

   > 运行游戏时出现以下错误，请帮我修复：
   > ```
   > Error: main.lua:45: attempt to call field 'newGrid' (a nil value)
   > ```

2. **确认 LÖVE 版本**：告诉AI你用的是哪个版本：

   > 我使用的是 LÖVE 11.5，请确保代码兼容这个版本。

3. **分段验证**：不要一次贴一大段代码。先写最小可运行版本，确认能跑，再逐步加功能。

## 坑位二：改完代码效果不对，但没报错

### 现象
代码运行了，窗口出来了，但效果和你想的不一样。比如蛇不移动、球穿过砖块、按键没反应等等。

### 原因
逻辑错误不会产生报错信息，但会让游戏行为不正确。

### 解决方案

1. **用 print 调试**：在可疑的位置加上 `print`，看变量值是否正确：

   ```lua
   print("蛇头位置:", snake[1].x, snake[1].y)
   print("方向:", direction.x, direction.y)
   print("分数:", score)
   ```

2. **描述具体现象**：告诉AI不是"代码有bug"，而是"具体什么行为不对"：

   > 蛇吃到食物后身体没有变长，但分数增加了。请检查碰撞后的蛇身增长逻辑。

3. **回退到上一个能工作的版本**：每改一步都保存一个备份。改出问题时，回到上一个能跑的版本，重新来。

## 坑位三：游戏越做越乱

### 现象
一个 `main.lua` 文件写了1000行，改一个功能其他功能就坏。

### 原因
所有逻辑混在一起，改A影响B。

### 解决方案

1. **拆文件**：按功能拆开

   ```
   main.lua          ← 入口，调用其他模块
   player.lua         ← 玩家相关的逻辑
   enemy.lua          ← 敌人相关的逻辑
   ui.lua             ← UI绘制
   ```

2. **用 local 限制作用域**：

   ```lua
   -- 不好：全局变量，到处都能改
   score = 0

   -- 好：用模块封装
   local M = {}
   local score = 0  -- 外部不能直接访问

   function M.get_score() return score end
   function M.add_score(n) score = score + n end
   return M
   ```

3. **每次只改一个功能，改完就测**。这是铁律。

## 坑位四：AI给你的代码风格不一致

### 现象
AI三次给了你三段代码，变量命名风格不同、缩进不同、逻辑写法也不同。

### 原因
每次回答AI都是重新生成，不会自动保持和你之前的代码一致。

### 解决方案

在提示词中明确代码风格：

> 请使用以下代码风格：
> - 变量名用 snake_case
> - 缩进用4个空格
> - 函数前面加注释说明用途
> - 和我现有代码保持一致的写法

或者让AI重构：

> 帮我统一这段代码的风格，和项目中的其他文件保持一致。以下是项目中的现有代码风格示例：

## 坑位五：性能问题

### 现象
游戏开始很流畅，玩一会儿越来越卡。

### 原因

1. **内存泄漏**：每帧创建新对象，旧的没有回收
2. **过度绘制**：画了屏幕上看不见的东西
3. **低效算法**：每帧遍历了大量数据

### 解决方案

1. **复用对象**：粒子用对象池，不要每帧 `table.insert` + `table.remove`

2. **减少每帧计算**：不变的值算一次存起来

3. **让AI帮你分析**：

   > 我的 LÖVE2D 游戏运行一段时间后变慢了。请在我的代码中找出可能的性能问题：
   > 1. 检查是否有每帧重复创建对象的地方
   > 2. 检查是否有不必要的全量遍历
   > 3. 检查图片加载是否在 update 中（应该在 load 中）

4. **用 LÖVE 的内置统计**：

   ```lua
   -- 在 love.draw 的末尾加上
   love.graphics.setColor(1, 1, 1)
   love.graphics.print("FPS: " .. love.timer.getFPS(), 10, 10)
   love.graphics.print("内存: " .. collectgarbage("count") .. " KB", 10, 30)
   ```

## 坑位六：路径和文件名问题

### 现象
在你的电脑上运行正常，给别人就找不到文件。

### 原因

1. Windows 不区分大小写，Linux 区分：`Image.png` 和 `image.png` 在 Windows 是同一个文件，在 Linux 是两个
2. 用了绝对路径：`C:\Users\xxx\game\...`
3. 中文路径或文件名导致编码问题

### 解决方案

1. **文件名全用小写，用下划线代替空格**
2. **用相对路径**：永远用 `"assets/images/player.png"` 而不是绝对路径
3. **避免中文文件名**

## 坑位七：版本管理混乱

### 现象
改了半天发现改错了，想回到之前能跑的版本，但找不到了。

### 解决方案

用 Git。哪怕你不会 Git 命令，Cursor 集成了 Git 操作：

```bash
# 初始化仓库
git init

# 每完成一个功能，提交一次
git add .
git commit -m "添加了食物生成逻辑"
```

**核心原则**：每次改完一个功能且测试通过，就提交一次。提交信息写清楚了改了什么。

---

## 坑位八：功能贪多嚼不烂

### 现象
你有一个绝妙的游戏想法，想一次性做完所有功能，结果什么都没做完。

### 解决方案

**MVP（最小可行产品）思路**：

1. 先做一个只能跑的版本——哪怕只有一个方块在屏幕上移动
2. 然后加一个功能——比如方块能跳
3. 再加一个功能——比如有障碍物
4. 每一步都能运行、都能玩

不要追求一步到位。游戏开发是一场马拉松，不是百米冲刺。

---

## 坑位九：AI上下文丢失

### 现象
你和AI聊了很久，突然AI给出的代码和你之前的完全不一致，好像忘记了之前说的。

### 原因
AI有上下文长度限制，超过限制后早期的内容会被遗忘。

### 解决方案

1. **定期让AI总结当前项目状态**：

   > 请总结一下当前游戏的所有功能和代码结构，我来确认一下。

2. **维护一个开发文档**：把每次修改记录下来，需要时贴给AI

3. **开始新对话时，先提供项目概述**：

   > 我在做一个 LÖVE2D 贪吃蛇游戏，目前功能包括：方向键控制、碰撞检测、分数系统、开始/结束界面。代码在 main.lua 中，以下是当前代码：（粘贴代码）
   >
   > 我现在想添加一个暂停功能。

## 坑位十：沉迷AI依赖

### 现象
什么都问AI，连改一个颜色值都要AI帮忙。离开AI完全写不了代码。

### 解决方案

AI是你的工具，不是你的替代。建议：

1. AI给的代码，至少通读一遍，大致理解它在做什么
2. 遇到简单的修改（改颜色、改数字），试着自己改
3. 每次AI修复了Bug后，想想如果下次遇到类似问题自己能不能修
4. 做完每个游戏后，写一个简短的总结：用了什么技术、踩了什么坑

不需要成为专业程序员，但理解一点代码会让你和AI的合作效率翻倍。

## 坑位十一：图片/音频加载失败

### 现象
运行游戏时弹出错误，常见报错信息：

```
Error: main.lua:12: Could not open file assets/images/player.png. Does not exist.
```

或

```
Error: main.lua:15: Cannot determine file type for assets/music/bgm.ogg.
```

没有图片的画面全是白色方块，没有声音的游戏一片安静。

### 原因

1. **文件路径写错了**：大小写不对、拼错了文件夹名、多了或少了子目录
2. **格式不支持**：LÖVE2D 图片支持 PNG/JPG/BMP/TGA，音频支持 OGG/WAV/MP3/FLAC。放了个 `.webp` 或 `.aac` 进去就会报错
3. **文件本身损坏**：下载或转码过程中文件损坏，能打开但 LÖVE 解析失败
4. **加载时机不对**：在 `love.load` 之外加载资源，导致路径解析出错

### 解决方案

1. **先确认文件存在且路径正确**：

   ```lua
   -- 在 love.load 里先用 io.open 检查文件是否存在
   local f = io.open("assets/images/player.png", "r")
   if f then
       f:close()
       player_img = love.graphics.newImage("assets/images/player.png")
   else
       print("警告：找不到图片 assets/images/player.png")
   end
   ```

2. **确认文件格式受支持**：图片用 PNG（支持透明），音频用 OGG（体积小、质量好）。不要用冷门格式。

3. **所有资源只在 `love.load` 中加载**，不要在 `love.update` 或 `love.draw` 中重复加载：

   ```lua
   function love.load()
       -- 所有资源在这里加载，只加载一次
       player_img = love.graphics.newImage("assets/images/player.png")
       bgm = love.audio.newSource("assets/music/bgm.ogg", "stream")
   end
   ```

## 坑位十二：理解债——代码能跑但你看不懂

### 现象
AI帮你写了很多代码，游戏跑得很爽，但你完全不知道怎么跑起来的。

### 原因
Vibe coding 最大的陷阱——"理解债"。代码是AI写的，不是你写的。社区称为"90天报应"：代码能跑≠你能维护。

### 解决方案

1. **定期让AI解释代码架构**：

   > 请总结当前游戏的代码结构和每个模块的功能

2. **维护一个开发日志**，记录每次修改了什么、为什么改

3. **每个功能模块至少通读一次**，理解输入→处理→输出

4. **用流程图辅助理解**：

   > 帮我画一个流程图，描述这段代码的逻辑

## 坑位十三：架构错了，越写越崩

### 现象
游戏功能越加越多，但每次加新功能都要改一大堆地方，改了A就坏B。

### 原因
没有在开工前想清楚架构。社区名言："Code is cheap in 2026, but wrong architecture is still expensive"

### 解决方案

1. **开工前画一个"模块图"**：哪些系统独立？哪些互相依赖？

2. **让AI帮你设计架构**：

   > 我想做一个XX游戏，请帮我设计代码架构，列出需要哪些模块、每个模块负责什么、模块之间如何通信

3. **先搭骨架，再填血肉**——先写空函数框架，确认架构合理后再填充逻辑

4. **用事件/消息解耦模块**，避免所有代码互相依赖

---

## 坑位十四：AI美术风格散架

### 现象
AI生成的图片每张单独看都好，放在一起像个大杂烩。

### 原因
没有约束。AI每次生成的风格都略有不同，不加约束就会散架。

### 解决方案

1. **固定调色板**：选7-8种颜色，要求所有素材只能在调色板内（参考 Void Balls 的7色调色板）

2. **固定描述词前缀**：每次生成时都带相同的风格描述词，如：

   > pixel art, 32x32, limited palette of 7 colors, dark fantasy style

3. **用一张参考图作为风格锚点**：先生成一张满意的，后续都以它为风格参考

4. **让AI帮你建立风格指南**：

   > 帮我创建一份游戏美术风格指南，包含调色板、线条粗细、风格关键词

## 坑位十五：游戏能跑但不好玩

### 现象
AI帮你做了所有功能，代码没问题，但玩起来就是没意思。

### 原因
AI懂代码，但不懂"好玩"。游戏手感、难度曲线、节奏感——这些是AI无法替代的。

### 解决方案

1. **自己玩15分钟**，记下所有"感觉不对"的地方
2. **找一个朋友玩**，不要教他/她，看哪里困惑
3. **对照 Game Feel 检查清单**（第7章）
4. **修复排名前三的问题**，不要加新功能
5. **记住：一个简单但好玩的游戏 > 一个复杂但无聊的游戏**

---

## 坑位十六：一个人闷头做，没人给反馈

### 现象
做了两个月游戏，发布后没人玩没人反馈。

### 原因
做游戏不是写日记——你要边做边分享。

### 解决方案

1. 做完第一个可玩版本就去 **r/aigamedev** 或 **r/vibecoding** 分享
2. **录制15秒GIF**展示最有意思的玩法
3. **写一段"build story"**："零基础，用XX工具，花了X天"
4. 在 **Discord** 游戏开发社区找同期开发者互相测试
5. **itch.io** 上开一个开发日志(devlog)

---

## 坑位十七：迭代节奏不对——改太多或太少

### 现象
要么一次改了5个功能结果不知道哪个出了bug，要么改一个颜色值也要问AI。

### 原因
没有建立正确的迭代节奏。

### 解决方案

1. **每次只改一个可验证的点**
2. **分层构建**：核心移动→关卡设计→计分/胜负→视觉效果→音效润色
3. **改完立刻运行测试**
4. **简单的改动**（改颜色、改数字）试着自己改
5. **复杂的改动**（新系统、新逻辑）让AI先给架构再填充

---

## 坑位十八：AI上下文窗口不够用了

### 现象
你的游戏代码越来越长，AI开始忘记之前的东西，给的代码和现有代码矛盾。

### 原因
AI有上下文长度限制。你的代码加上对话历史超过限制后，之前的内容就被截断。

### 解决方案（比坑位九更深入的版本）

1. **维护一个 ARCHITECTURE.md 文件**记录项目结构，每次新对话先贴这个
2. **把大文件拆成小模块**，每次只给AI相关的模块代码，而不是全部代码
3. **使用"总结-继续"策略**：对话快到上限时，让AI总结当前状态，开新对话继续
4. **用 Cursor/Pi 等IDE集成的AI助手**，它们能自动读取项目文件
5. **关键：把每个模块的接口**（函数签名+注释）单独维护，让AI只需看接口就能理解用法
