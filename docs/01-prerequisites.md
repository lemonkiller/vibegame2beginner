# 第1章 前置认知

> 在开始之前,让我们先搞清楚几件重要的事。

## 你不需要会写代码

这是一个需要反复强调的事实:**在2026年,你不需要会写代码也能做游戏。**

想想你用手机点外卖的过程:你不需要知道外卖APP的后台是怎么写的,你只需要在屏幕上点几下,饭就送到了。现在做游戏也变成这样了--你告诉AI你想要什么,AI帮你把代码写出来。

但这不意味着你什么都不用学。你需要学的是:

| 你需要学什么 | 你不需要学什么 |
|-------------|--------------|
| 用自然语言清楚描述你的想法 | 记忆编程语法 |
| 看到效果后判断"哪里不对" | 从零写代码 |
| 把复杂需求拆成小步骤 | 搭建开发环境 |
| 把错误信息复制给AI帮忙修 | 手动调Bug |

简单来说,你的角色从"写代码的人"变成了"指挥AI写代码的人"。就像从"亲自炒菜"变成了"告诉厨师你想要什么口味"。

## AI能做什么

### 非常擅长的事

- **写小型游戏的完整代码**:贪吃蛇、俄罗斯方块、打砖块这类经典小游戏,AI可以一次性生成可运行的版本
- **帮你理解代码**:把看不懂的代码丢给AI,它能用白话解释给你听
- **修复Bug**:把报错信息复制给AI,它能帮你找到问题并修复
- **生成游戏素材**:AI可以生成像素画、音乐、音效
- **翻译需求为代码**:你用中文说"角色跳起来的时候有个阴影跟着缩放",AI就帮你写出来

### 能做但需要你盯着的事

- **中大型游戏**:AI可以帮忙写各个模块,但你需要自己把控整体结构和进度
- **性能优化**:AI给出的代码可能不是最优的,需要你测试后反馈问题
- **创意设计**:AI可以给建议,但游戏的"灵魂"还是你的

### 目前做不到的事

- **一键生成3A大作**:AI不是魔法,复杂游戏还是需要大量人工打磨
- **完全理解你的意图**:AI有时候会"听错话",需要你反复纠正
- **保证代码100%正确**:AI生成的代码可能有问题,需要你测试验证
- **替代你对游戏体验的判断**:好不好玩,只有玩了才知道

## 游戏复杂度地图

做游戏之前,先搞清楚你的想法落在哪个难度区间。下面这张地图用两个维度衡量一个游戏的复杂度:

- **横向**:系统复杂度(有多少个互相影响的系统--移动、碰撞、计分、UI、音效各算一个)
- **纵向**:内容复杂度(需要多少手工制作的内容--关卡、对话、素材、剧情)

```
内容多 ↑
      │                    ● RPG对话冒险
      │
      │              ● 视觉小说
      │
      │         ● 打砖块
      │
      │   ● 贪吃蛇              ● 平台跳跃
      │
      │ ● 2048   ● 打地鼠
内容少 ┼────────────────────────────→ 系统多
      系统少                     系统多
```

### 四个区域解读

| 区域 | 特点 | 典型游戏 | AI怎么帮你 | 你需要做什么 |
|------|------|---------|----------|------------|
| **左下角** 系统少+内容少 | 纯机制小游戏 | 2048、打地鼠、Flappy Bird | 一次生成,直接可玩 | 测试并微调 |
| **左上角** 系统少+内容多 | 故事驱动 | 视觉小说、文字冒险 | 帮你批量生成对话和剧情 | 把控故事方向和节奏 |
| **右下角** 系统多+内容少 | 机制复杂 | 平台跳跃、塔防、飞行射击 | 分模块写代码 | 先想清楚架构再动手 |
| **右上角** 系统多+内容多 | 大项目 | RPG、策略游戏 | 各模块分别写 | 需要项目管理能力 |

**经验法则**:你的第一个游戏应该在左下角。本手册的三个游戏完成后,你可以向左上角(做故事)或右下角(做机制)扩展。右上角等你学完第12章"避坑指南"再说。

### 升级路径

```
路径A(机制派):贪吃蛇 → 打砖块 → 平台跳跃 → 动作RPG     向右走📈
路径B(故事派):贪吃蛇 → RPG对话冒险 → 视觉小说 → 开放世界RPG  向上走📈
路径C(全栈派):贪吃蛇 → 打砖块 → RPG → 策略游戏     对角线走📈📈
```

选哪条路径取决于你喜欢"做机制"还是"做内容"。没有对错,只有适合。

### 系统数量速查

一个更简单的判断方式--数你游戏里有几个独立的"系统":

| 系统数量 | 举例 | AI单次生成能力 | 你的参与度 |
|---------|------|--------------|----------|
| 1-3个 | 贪吃蛇(移动+碰撞+计分) | 一次搞定 | 只需测试 |
| 4-6个 | 打砖块(+反弹+生命+粒子) | 分2-3次做 | 需要拆解+测试 |
| 7-10个 | RPG(+对话+战斗+道具+地图) | 分5+次做 | 需要架构思维 |
| 10+个 | 策略游戏(+AI+经济+外交...) | 需要多Agent协作 | 需要专业项目管理 |

**本手册选择路线B**,使用 LÖVE2D + AI编程助手的方式。原因:

1. 你能看到AI生成的代码,理解原理,而不是黑箱操作
2. 灵活度够高,从小游戏到中等复杂度的2D游戏都能做

4. 后期如果想走路线C,学到的知识可以直接迁移

## 迭代哲学:涟漪式开发

传统做游戏像盖房子--先画图纸,再打地基,再砌墙,改一面墙可能要拆半栋楼。AI做游戏像捏泥巴--揉一下看一眼,不满意再揉,每次揉只要30秒。

这彻底改变了开发节奏:**从"先想清楚再做"变成"先做出来再想清楚"。**

### 传统开发 vs AI涟漪式开发

| 维度 | 传统开发 | AI涟漪式开发 |
|------|---------|------------|
| 一个试错循环 | 2-4小时 | 2-5分钟 |
| 改主意成本 | 很高(重写代码) | 极低(改一句话重新生成) |
| 规划深度 | 必须先想清楚 | 先做最糙的版本看效果 |
| 失败心态 | "又浪费一下午" | "才3分钟,换个方案试试" |
| 最佳节奏 | 分阶段(周/月) | 分涟漪(分钟级) |

### 涟漪式开发的三条铁律

**铁律一:每圈涟漪必须可玩。** 不是"写了一半的代码",是"能跑起来能看到效果的最小版本"。哪怕只是一个方块在屏幕上弹来弹去。

**铁律二:每圈涟漪只加一个变化。** 同时加碰撞+加计分+加界面 = 三圈涟漪揉成一团,出问题不知道哪圈破了。一圈一圈来。

**铁律三:涟漪不完美也没关系。** 第一圈做出来的东西一定糙。别纠结。下一圈涟漪会覆盖它、改善它。**完美主义是涟漪式开发最大的敌人。**

### 涟漪可视化

```
第1圈  角色能动        ╭─╮
第2圈  +碰撞检测      ╭─╮╭─╮
第3圈  +计分系统      ╭─╮╭─╮╭─╮
第4圈  +游戏界面    ╭─╮╭─╮╭─╮╭─╮
第5圈  +手感润色  ╭─╮╭─╮╭─╮╭─╮╭─╮
```

每一圈都覆盖在前一圈之上,越来越完整。每一圈结束,你都有一版可以玩的游戏。

- 做出3A级别的游戏(这需要专业团队)

- 一次性就做出完美的游戏(迭代是游戏开发的核心)

**但这些都不重要**--重要的是你可以从今天就开始做你的第一个游戏了。
