# 附录F AI多智能体协作开发

> 2026年最兴奋的事：你不再是一个人对着一台AI死磕，而是像开公司一样——有人设计、有人干活、有人挑毛病。

**数据时效性说明**：本附录中的工具信息、工作流案例截至 **2026年4月**。AI工具迭代飞快，建议查看各工具官网获取最新功能。

---

## 引言：2026年的新玩法

还记得你第一次用AI编程的感觉吗？一个对话框，你打字，AI回代码，粘贴过去跑一跑——完事了。

但这套玩法有个明显上限：**项目一大，一个AI就忙不过来了**。它既要想架构，又要写代码，还要帮你查bug，最后把自己绕晕了，输出质量直线下降。

2026年社区出现了一个新范式：**多智能体协作**（Multi-Agent Collaboration）。

什么意思呢？打个比方——

> 开餐厅不是让一个人又做饭又端菜又收钱又擦桌子，那是超人干的事。正常餐厅有厨师、服务员、收银员、保洁员，各司其职，效率翻倍。

多智能体协作就是同样的道理：**同时开多个AI对话，让每个AI专注于不同任务**。一个负责设计架构，一个负责写代码，一个负责测试，各管一摊，互不干扰。

这不是什么高深概念，你只需要学会"分工"这个常识。

---

## 一、什么是多智能体协作？

### 最简单的解释

就是**同时开好几个AI对话窗口，每个窗口里让AI扮演不同角色**。

这不是让你学什么新框架、新语言。你以前怎么跟AI聊天，现在还是怎么聊——只不过从"一对一"变成了"一对多"。

### 现实案例：Void Balls

2026年初，一个叫 **Void Balls** 的游戏项目在社区里炸了锅。它用 **8个并行Agent** 做了这么一件事：

| 指标 | 数据 |
|------|------|
| 代码量 | 29,000行 C# |
| 脚本文件 | 173个 |
| 自动测试 | 88个 |
| 开发时间 | **10天** |

8个Agent，分工明确，10天交付了一个3A级的游戏原型。这如果是单人单AI，可能得做3个月。

当然，Void Balls 用的是 Unity（不是 LÖVE2D），但**协作思路是完全通用的**——分工会带来效率飞跃，跟具体用什么引擎无关。

### 单Agent vs 多Agent 效率对比

| 对比维度 | 单Agent | 多Agent（3+） |
|---------|---------|--------------|
| 项目规模 | 适合做1-2个文件的小游戏 | 可以驾驭几十个文件的中大型项目 |
| 代码质量 | 后期容易"迷路"，越改越乱 | 每个Agent专注一块，代码更清晰 |
| 迭代速度 | 改一个东西可能牵一发动全身 | 各模块独立迭代，互不干扰 |
| 出错率 | 一个Agent脑子装太多，容易自相矛盾 | 有专门的"检查员"把关 |
| 上手难度 | ⭐ 最简单 | ⭐⭐⭐ 需要一些组织能力 |

**关键结论**：小项目用单Agent，大项目用多Agent。不是越复杂越好，而是**按需选择**。

## 二、角色分工（用餐厅比喻）

想象你开了一家餐厅，每个AI Agent就是一个员工：

### 🍳 架构师（Architect）—— 餐厅的行政主厨

**角色定位**：不抄菜，但决定菜单结构、厨房分区、动线设计。

**具体职责**：
- 设计游戏的整体代码结构（哪些模块、怎么通信）
- 规划目录结构和文件分工
- 定义模块之间的接口（谁管谁要数据，格式是什么）
- 在项目开始前"画蓝图"

**典型提示词示例**：
```
请帮我设计一个 LÖVE2D 贪吃蛇游戏的代码架构，要求：
- 把游戏分成独立的模块（蛇、食物、分数、界面等）
- 每个模块只管自己的事，模块间用消息通信
- 画出目录结构，每个文件负责什么
- 说明模块之间的调用关系
```

**适合阶段**：✅ 项目开始时（第一步）
**不适合**：❌ 已经在写代码了再找架构师——那时候改动成本太高

---

### 🔧 实现者（Implementer）—— 餐厅的炒菜厨师

**角色定位**：拿着菜单（架构），把每道菜做出来。

**具体职责**：
- 按照架构师设计的模块结构，编写具体功能代码
- 遵循接口规范，不越界改别的模块
- 实现游戏逻辑、渲染、输入处理等

**典型提示词示例**：
```
根据以下架构设计，帮我实现 LÖVE2D 贪吃蛇的"蛇"模块：
- 文件名：snake.lua
- 蛇用链表存储身体各节
- 提供 update(dt) 和 draw() 方法
- 提供 changeDirection(dir) 接口给外部调用
- 碰到自己身体时触发 onSelfCollision 回调
- 不要引用其他模块的内部变量
```

**适合阶段**：✅ 架构设计完成后
**不适合**：❌ 还没有架构就直接开写——写着写着就乱了

---

### ⚖️ 平衡师（Balancer）—— 餐厅的口味调校师

**角色定位**：菜做出来了，但咸了淡了？辣了甜了？他来调。

**具体职责**：
- 调整游戏数值（敌人血量、移动速度、伤害值等）
- 设计难度曲线（越往后越难，但不能太离谱）
- 平衡游戏节奏（太简单无聊，太难劝退）
- 让游戏"好玩"

**典型提示词示例**：
```
帮我调整这个 LÖVE2D 射击游戏的数值，当前问题是：
- 前3关太简单，敌人一枪一个，无聊
- 第4关突然变难，玩家3秒就死了
- 武器升级后太强，完全碾压
请帮我设计一个平滑的难度曲线，每关敌人血量和速度递增的数值表，
保证每关难度提升约10-15%，不会出现断崖式跳跃。
```

**适合阶段**：✅ 核心玩法跑通后、打磨阶段
**不适合**：❌ 游戏还跑不起来时就调数值——没意义

---

### 🔍 测试员（Tester）—— 餐厅的品控

**角色定位**：菜上桌前他先尝，有虫子绝对过不了他这关。

**具体职责**：
- 为关键函数写单元测试
- 检查边界情况（敌人跑到屏幕外怎么办？分数溢出怎么办？）
- 验证模块间的接口是否正确对接
- 找出实现者没注意到的bug

**典型提示词示例**：
```
请为以下 LÖVE2D 碰撞检测函数写测试用例：
- 正常碰撞（两个矩形重叠）
- 边缘碰撞（刚好挨着但不重叠）
- 完全包含（小矩形在大矩形里）
- 负坐标情况
- 超大坐标情况
- 零宽高矩形
要求每个测试用例说明：输入是什么、期望输出是什么、为什么这样测。
```

**适合阶段**：✅ 每个模块实现完成后
**不适合**：❌ 架构设计阶段——那时候还没代码可以测

---

### 🎨 美术指导（Art Director）—— 餐厅的摆盘师

**角色定位**：菜好吃不够，还得好看。他负责视觉风格统一。

**具体职责**：
- 生成游戏素材（角色、背景、UI元素）
- 确保所有素材风格统一（像素风？卡通风？写实风？）
- 设计调色板和视觉规范
- 用AI图像工具（如 Midjourney、DALL-E）生成素材

**典型提示词示例**：
```
帮我为一个 LÖVE2D 像素风RPG生成以下素材描述：
- 主角：16x16像素，4方向行走动画，每方向3帧
- 怪物：3种，同风格，区分度要高
- 地面瓦片：16x16，草地/泥土/石头/水4种
- UI框架：对话框、血条、物品栏
整体风格：暖色调像素风，参考《星露谷物语》但更Q版。
请给出每个素材的具体像素描述，我拿去AI画图用。
```

**适合阶段**：✅ 有了基本玩法原型后（先能用方块跑起来，再换美术）
**不适合**：❌ 游戏还没做就先画图——可能做完发现架构不对，白画了

---

### 角色总览表

| 角色 | 餐厅比喻 | 负责什么 | 提示词开头 | 适合阶段 |
|------|---------|---------|-----------|---------|
| 🍳 架构师 | 行政主厨 | 代码结构、模块划分、通信方式 | "请帮我设计XX游戏的代码架构..." | 项目开始 |
| 🔧 实现者 | 炒菜厨师 | 按架构写具体功能代码 | "根据以下架构，帮我实现XX模块..." | 架构完成后 |
| ⚖️ 平衡师 | 口味调校 | 数值、难度曲线、游戏节奏 | "帮我调整XX数值，让游戏不太简单也不太难..." | 核心玩法跑通后 |
| 🔍 测试员 | 品控 | 写测试、检查边界情况 | "帮我为XX函数写测试用例..." | 模块实现后 |
| 🎨 美术指导 | 摆盘师 | 生成美术素材、确保风格统一 | "帮我生成以下游戏素材，风格要求..." | 有原型后 |

## 三、新手怎么开始用？

别急着一上来就8个Agent齐飞——那会手忙脚乱。我们分三步走：

### 第一步：先不要用 🚫

**对，你没看错。先别用多Agent。**

新手最重要的事是什么？是用**一个AI，从头到尾做完一个完整小游戏**。哪怕只有100行代码，哪怕很丑很简陋，但你要经历"从0到1"的完整流程——

- 怎么提需求
- 怎么让AI写代码
- 怎么看代码有没有问题
- 怎么迭代改进
- 怎么打包发布

这些基本功，单Agent就能练。**先会用一把锤子，再学用一整个工具箱。**

> 🎯 **建议**：用单AI至少做完本手册的贪吃蛇（第三章）和打砖块（第四章），再考虑多Agent。

---

### 第二步：用2个角色 👥

当你能顺畅地完成小游戏后，试试这个最简单的多Agent组合：

| 角色1：写代码的人 | 角色2：看代码的人 |
|-----------------|-----------------|
| 写新功能、改bug | 检查写出来的代码有没有问题 |

具体操作：

1. **第一个对话窗口**：「请帮我实现XX功能」，把AI写的代码放到项目里
2. **第二个对话窗口**：把那段代码贴进去，说「请帮我审查这段代码，找出潜在的bug、性能问题、和可以改进的地方」

这就是"自己写 + 别人查"的基本模式。非常简单，但**非常有效**——很多你忽略的问题，另一个视角的AI一眼就能看到。

> 💡 **小技巧**：第二个Agent（审查者）的提示词可以这样写：
> ```
> 请审查以下 LÖVE2D 代码，重点关注：
> 1. 有没有明显的bug
> 2. 有没有性能问题（比如每帧创建新表）
> 3. 有没有违反已有的模块接口
> 4. 代码风格是否一致
> 5. 有没有遗漏的边界情况
> ```

---

### 第三步：按需扩展 🔄

当你的项目变大（比如开始做RPG，几十个文件），2个Agent不够用了，按需加人：

- 发现架构混乱？➜ **加一个架构师Agent**
- 发现数值不对？➜ **加一个平衡师Agent**
- 发现频繁出bug？➜ **加一个测试员Agent**
- 需要美术素材？➜ **加一个美术指导Agent**

**原则：缺什么补什么，不要一上来就凑齐5个。** 每多一个Agent，你的协调成本也在增加。

> 🎯 **经验法则**：如果你发现自己频繁在一个对话里说"等等你别管这个，你管那个"——就该分出新的Agent了。

## 四、实用工具链

2026年支持多智能体开发的主要工具：

### 🏆 Pi Coding Agent（本项目推荐）

- **原生支持子智能体**（subagent），可以直接在对话里派活给专门的Agent
- 支持并行（PARALLEL）和链式（CHAIN）两种协作模式
- 内置代码搜索、LSP导航，Agent之间可以共享项目上下文
- **最适合**：本手册读者，因为本手册全程用的就是它

### 🔲 Cursor

- **多Tab模式**：开多个Tab，每个Tab一个AI对话，手动分工
- Composer模式可以同时编辑多个文件
- 需要手动在Tab之间复制粘贴来"传递信息"
- **最适合**：已经习惯 Cursor 工作流的开发者

### 🔄 Claude Code

- **并行Agent模式**：可以启动多个并行Agent实例
- 每个实例有自己的工作目录或焦点
- 支持 git worktree 隔离，避免文件冲突
- **最适合**：命令行习惯的开发者

### 📦 Codex（OpenAI）

- **任务拆分模式**：可以把大任务自动拆成小任务
- 每个小任务由独立Agent完成
- 输出结构化，方便汇总
- **最适合**：结构化、可拆分的任务

## 五、注意事项

多Agent虽好，但有几个坑你必须知道：

### 🚨 1. 不要让多个AI改同一个文件

这是**最重要的一条**。

想象两个人同时在同一份文档上打字——你打你的，我打我的，最后合并就是一锅粥。多AI改同一个文件也会这样：Agent A加了函数，Agent B删了同一行，最后谁也不认识谁。

**解决方案**：
- 每个Agent只负责自己的文件/模块
- 如果确实需要改公共文件，一个改完确认了，另一个再改
- 用Git管理版本，出问题可以回退

### 🚨 2. 每个Agent要有明确的边界和接口

就像餐厅里厨师不管收钱，收银员不管做菜。**边界模糊就会互相踩脚**。

**怎么做**：
- 架构师先定义好每个模块的接口（有哪些函数、参数是什么、返回什么）
- 实现者严格按接口写代码，不越界
- 如果发现接口不合理，回到架构师修改——不要自己偷偷改

### 🚨 3. 架构师角色必须先于实现者

先有蓝图再盖楼，这是常识。但很多人一上来就让AI写代码，写着写着发现结构不对，推倒重来，浪费时间。

**正确顺序**：
1. 架构师设计 📋
2. 你审查架构 👀
3. 实现者按架构写代码 🔧
4. 测试员检查代码 🔍

永远不要跳过第1-2步。

### 🚨 4. 人工审查仍然是必须的

AI再聪明也不是人。它可以写出"看起来对"的代码，但逻辑可能完全反了。它可以设计出"看起来合理"的架构，但可能遗漏关键场景。

**你是最后的把关人**。每个Agent的输出，你至少要扫一眼：

- 代码能不能跑？
- 架构合不合理？
- 测试用例有没有遗漏？
- 数值调了之后好不好玩？

> 💡 **记住**：多Agent是帮你干活的工具，不是替你思考的替代品。最终决策权永远在你手里。

### 🚨 5. 沟通成本会随Agent数量增加

2个Agent的管理成本很低。但5个、8个呢？你需要——

- 记住每个Agent在做什么
- 确保它们之间的接口一致
- 把一个Agent的输出"搬运"给另一个Agent
- 检查各Agent的输出有没有矛盾

这不是不用动脑了，而是**换了一种动脑方式**——从"写代码"变成"管团队"。

## 六、一个完整示例：LÖVE2D贪吃蛇用2-Agent

让我们用最简单的2-Agent模式（架构师 + 实现者），走一遍完整流程。

### Step 1：架构师Agent 设计模块结构

**提示词**：
```
请帮我设计一个 LÖVE2D 贪吃蛇游戏的代码架构，要求：
1. 模块化设计，每个模块一个文件
2. 模块之间通过明确接口通信，不互相访问内部变量
3. 画出目录结构
4. 列出每个模块的接口（函数名、参数、返回值）
5. 说明模块间的依赖关系

游戏需求：
- 经典贪吃蛇玩法
- 方向键控制
- 吃到食物蛇变长、加分
- 撞墙或撞自己游戏结束
- 有开始界面、游戏界面、结束界面
```

**预期输出**（架构师给出的方案）：

```
目录结构：
├── main.lua          -- 入口，加载各模块，管理游戏状态
├── snake.lua         -- 蛇的逻辑：移动、生长、碰撞检测
├── food.lua          -- 食物逻辑：随机生成、被吃检测
├── score.lua         -- 分数管理：计分、显示
├── grid.lua          -- 网格系统：坐标转换、边界检查
└── ui.lua            -- 界面管理：开始/游戏/结束三个界面

模块接口：

snake.lua:
  - Snake:create(gridX, gridY)   -- 创建蛇，初始位置
  - Snake:update(dt)             -- 更新蛇的位置
  - Snake:draw(cellSize)         -- 绘制蛇
  - Snake:setDirection(dir)      -- 设置方向（"up"/"down"/"left"/"right"）
  - Snake:getHead()              -- 返回蛇头坐标 {x, y}
  - Snake:grow()                 -- 蛇增长一格
  - Snake:checkSelfCollision()   -- 检测是否撞到自己，返回bool
  - Snake:getLength()            -- 返回蛇的长度

food.lua:
  - Food:create(grid)            -- 创建食物，传入grid模块
  - Food:respawn()               -- 重新随机生成位置
  - Food:draw(cellSize)          -- 绘制食物
  - Food:getPosition()           -- 返回食物坐标 {x, y}

grid.lua:
  - Grid:create(cols, rows)      -- 创建网格
  - Grid:isInside(x, y)         -- 检查坐标是否在边界内
  - Grid:randomCell()            -- 返回随机格子的坐标

score.lua:
  - Score:create()               -- 创建计分器
  - Score:add(points)            -- 加分
  - Score:draw(x, y)            -- 在指定位置绘制分数
  - Score:getScore()             -- 返回当前分数

ui.lua:
  - UI:drawStartScreen()         -- 绘制开始界面
  - UI:drawGameOverScreen(score) -- 绘制结束界面
  - UI:drawGameUI(score)         -- 绘制游戏中UI（分数等）

模块依赖关系：
main.lua → snake, food, score, grid, ui
snake.lua → grid（检查边界）
food.lua → grid（随机位置、避开蛇身）
score.lua → 无依赖
ui.lua → score（显示分数）
```

### Step 2：实现者Agent 分别实现各模块

现在拿着架构师的蓝图，让实现者Agent一个一个模块写。

**提示词（以 snake.lua 为例）**：
```
请根据以下架构规范，帮我实现 LÖVE2D 贪吃蛇的 snake 模块。

要求严格遵循以下接口：
- Snake:create(gridX, gridY)   -- 创建蛇，初始位置
- Snake:update(dt)             -- 更新蛇的位置
- Snake:draw(cellSize)         -- 绘制蛇
- Snake:setDirection(dir)      -- 设置方向
- Snake:getHead()              -- 返回蛇头坐标
- Snake:grow()                 -- 蛇增长一格
- Snake:checkSelfCollision()   -- 检测是否撞到自己
- Snake:getLength()            -- 返回蛇的长度

注意事项：
- 蛇的身体用 table 存储各节坐标
- 每次update先移动身体，再更新蛇头
- grow时不删除尾巴（正常移动时删除尾巴）
- 不允许180度转向（不能从左直接变右）
- 不要引用其他模块的内部变量
- 代码加中文注释
```

**预期输出**：实现者Agent会按照以上接口规范，生成包含以下关键逻辑的 snake.lua：

- 蛇身用 table 存储各节坐标，body[1] 为蛇头
- `setDirection()` 会检查是否为反方向（不允许180度转向）
- `update()` 先移动身体（头部插入新位置、尾部删除），再判断是否增长
- `grow()` 设置标记，下次 update 时不删除尾巴
- `checkSelfCollision()` 遍历身体各节与蛇头坐标比较
- `draw()` 蛇头绿色、身体暗绿色，用 rectangle 绘制

对其他模块（food.lua、grid.lua、score.lua、ui.lua、main.lua），用同样的方式——拿着架构师的接口规范，让实现者Agent分别写。

**关键心得**：2-Agent（架构师 + 实现者）加上最后的测试审查，就能覆盖小项目的完整流程。等你觉得2个不够用了，再参考前面的角色表加人。

## 小结

多智能体协作不是什么高不可攀的技术，它的本质就是**分工**——你已经在生活中这么做了，现在只是把同样的常识用到AI编程里。

记住三件事：

1. **新手先别用**——单AI做完一两个游戏再说
2. **从2个角色开始**——写代码 + 审查代码，最简单也最有效
3. **按需扩展**——项目变大再加人，别为了酷而酷

2026年AI编程的核心不是"AI多强"，而是"你会不会组织AI团队"。这本手册教你的LÖVE2D知识和提示词技巧，是基础中的基础——地基打好了，上面的楼想盖几层都行。
